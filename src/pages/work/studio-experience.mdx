        <h4>Bridging the Interaction Gap</h4>
        <p>While the visual approach to workflow configuration tested well conceptually, early validation revealed a significant gap between user expectation and system capability.</p>
        <ul>
          <li><strong>The Drag-and-Drop Expectation:</strong> In early prototypes, users intuitively tried to <strong>drag and drop nodes</strong> on the visualization canvas to reorganize the flow, a standard interaction pattern that the low-code platform did not natively support.</li>
          <li><strong>The Pivot:</strong> Recognizing this friction, the design was pivoted to a <strong>visualization-first</strong> behavior. We rebuilt the interaction model to support <strong>direct manipulation</strong> where possible and added clear visual cues for selecting and editing tasks where drag-and-drop was technically constrained.</li>
          <li><strong>Validation:</strong> Despite the initial friction, users reported high confidence in the overall model, validating that the <strong>visual representation</strong> was the correct approach for handling complex case management logic.</li>
        </ul>

  <ProcessSection title="Process: Defining the Hierarchy">
    <TwoCol sticky={true} frame={true}>
      <div slot="text">
        <h4>Determining Logical Scope</h4>
        <p>Early feedback indicated that users found it difficult to determine the appropriate location for automation logic that applied across multiple case types but required local variations.</p>
        <ul>
          <li><strong>Card Sort Results:</strong> To define the system architecture, a card sorting activity was conducted with seven participants. The results showed that users intuitively grouped automation needs by <strong>scope (global vs. case-specific)</strong> rather than by the type of trigger or action.</li>
          <li><strong>The Decision:</strong> These findings led to the development of a <strong>hierarchical inheritance model</strong>. Instead of manual overrides, the system enforces a strict precedence logic: specific rules (case type level) always <strong>take priority</strong> over general rules (global level).</li>
          <li><strong>Consistency:</strong> This logic was aligned with the existing behavior for user-defined data fields and forms, ensuring that the <strong>"lowest level wins" pattern</strong> remained consistent across the entire Studio configuration experience.</li>
        </ul>
      </div>
